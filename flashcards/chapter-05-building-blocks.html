Why is getLast(vector) unsafe even though Vector is thread-safe?
Vector guarantees each method is atomic, but getLast is a compound action: size() then get(lastIndex). Between the two calls another thread can shrink the vector, so get throws ArrayIndexOutOfBoundsException.

Fix: client-side lock on the collection itself. 
synchronized (list) {
    int lastIndex = list.size() - 1;
    return list.get(lastIndex);
}
What makes synchronized-collection iterators "fail-fast," and why is CME only an early-warning indicator?
Iterators track a modification count on the collection. If hasNext or next sees it change since iteration began, they throw ConcurrentModificationException.

The check is performed without synchronization (a deliberate perf tradeoff), so it can miss modifications due to stale reads. CME catches problems on a good-faith-effort basis only — don't rely on it as a safety net.
What is a "hidden iterator" and where do they lurk?
Any code path that iterates a collection without an explicit for loop. Common culprits:

• toString() — triggered by "msg: " + set via StringBuilder.append(Object)
• hashCode() / equals() when the collection is an element or key of another
• containsAll, removeAll, retainAll
• Collection copy constructors that take another collection

A debug println without the collection's lock can throw CME. Encapsulate the collection (e.g. wrap with synchronizedSet) so you can't forget.
Why is ConcurrentHashMap usually a drop-in replacement for Hashtable/synchronizedMap?
It uses lock striping — many readers can run concurrently with each other and with a bounded number of writers, so throughput scales with cores instead of being pinned by one lock.

Its iterators are weakly consistent (no CME), and it offers better scalability with little downside — unless you specifically need to lock the map for exclusive access.
What four compound operations does ConcurrentMap add, and why do they exist? 
V putIfAbsent(K, V)
boolean remove(K, V)                        // remove-if-equal
boolean replace(K, V old, V new)            // replace-if-equal
V replace(K, V new)                         // replace-if-present
They exist because ConcurrentHashMap can't be locked from outside to build your own atomic operations — so it bakes the common compound actions into the interface itself.
When is CopyOnWriteArrayList the right choice, and when is it a disaster?
Right when iteration vastly outnumbers mutation (the canonical case is event-listener lists). No locking or CME during iteration — readers see the snapshot as of when they started.

Disaster when writes are frequent: every mutation allocates and copies the entire backing array, so an N-element list is O(N) to mutate.
What problems does the producer-consumer pattern with a BlockingQueue solve?
Decoupling — producers and consumers don't know each other's count or identity.

Flow control — a full bounded queue blocks producers, naturally throttling the system.

Resource management — bounded queues prevent OOM when consumers fall behind.

Performance — CPU-bound consumers and I/O-bound producers run concurrently for better overall throughput.

What is SynchronousQueue and when should you use it?
A "queue" with no storage — it maintains lists of waiting producers/consumers and hands work off directly. put blocks until a consumer is ready; take blocks until a producer is ready.

Benefits: lower latency (no staging step) and natural backpressure (the producer knows the instant a consumer accepted the work).

Only suitable when consumers are plentiful enough to nearly always be ready for handoff.
What is serial thread confinement and how does a BlockingQueue enable it?
A pattern for sharing mutable objects: ownership transfers from one thread to another via a safe-publication handoff, and the original owner never touches the object again. The receiver then has exclusive access and can mutate freely.

A BlockingQueue provides the safe publication automatically, so producers can publish mutable work items, consumers can take them, and nothing needs further synchronization — provided the producer honors the "don't touch it after handoff" rule.

Object pools work the same way.
What is work stealing, and why does it scale better than a shared producer-consumer queue?
Each worker has its own Deque. It works on its own deque from the head (no contention). When empty, it steals work from the tail of another worker's deque — the far end, so the owner (head) and the thief (tail) rarely contend.

Perfect when consumers are also producers (processing one unit yields more work): web crawlers, graph traversal, GC heap marking.
What does it mean that interruption is "cooperative," and what are your two options when you catch InterruptedException?
Thread.interrupt() just sets a boolean flag. Nothing forces the target thread to stop — blocking library methods check the flag and throw; pure-CPU loops must poll. It's a polite request, typically used for cancellation.

When you catch InterruptedException in your own code, two acceptable moves:

1. Propagate the exception to a caller that knows what to do (or just don't catch it).
2. Restore the flag if you can't throw: 
catch (InterruptedException e) {
    Thread.currentThread().interrupt();
}
Never swallow it — that destroys evidence of the interrupt and breaks cancellation for code up-stack.
What is a synchronizer, and what are the four main types covered in chapter 5?
A synchronizer is any object that coordinates thread control flow based on its state — it encapsulates state, provides methods to mutate it, and provides methods to wait for the state to become something useful.

Latches (CountDownLatch) — one-shot gate that opens once.
Futures (FutureTask) — latch for a result-bearing computation.
Semaphores (Semaphore) — control how many threads may pass.
Barriers (CyclicBarrier, Exchanger) — rendezvous points for N threads.

BlockingQueues also count as synchronizers.
What's the difference between a latch and a barrier?
"Latches are for waiting for events; barriers are for waiting for other threads."
Latch — one-shot gate. Once the terminal state is reached, it stays open forever. Typical use: delay progress until N events have occurred.

Barrier — reusable rendezvous. Every participant must arrive before any is released. Typical use: step-by-step parallel algorithms where all workers must finish step k before anyone starts step k+1.
Why does TestHarness use both a starting gate and an ending gate?
Starting gate (count = 1): every worker await()s it. Without it, threads created earlier get a head start, and the mix of active threads changes over time — bad for measuring concurrent behavior. The master releases everyone at once via countDown().

Ending gate (count = N): each worker calls countDown() on completion. The master await()s it to block until the last worker finishes, so the elapsed time captures the whole group.
What is FutureTask, and why does Future.get() constitute safe publication?
FutureTask<V> is a result-bearing computation built on Callable<V>. Three states: waiting, running, completed (by normal return, exception, or cancellation). It acts like a latch — get() returns immediately if completed, else blocks.

The spec guarantees Future.get() provides safe publication of the result from the computing thread to the retrieving thread — no extra synchronization needed to see a fully constructed result.
What does launderThrowable do, and why does it exist?
Future.get() wraps everything a task throws — checked exceptions, unchecked exceptions, and Errors — in an ExecutionException whose getCause() is a Throwable. Awkward to handle.

launderThrowable unwraps it: 
if (t instanceof RuntimeException) return (RuntimeException) t;
else if (t instanceof Error) throw (Error) t;
else throw new IllegalStateException("Not unchecked", t);
Callers first test for and rethrow any known checked exceptions from their Callable, then hand the rest to launderThrowable and throw its return value.
What does a counting semaphore do, and what's one surprising property?
A Semaphore(N) manages N virtual permits. acquire() takes one (blocking if none available); release() returns one. Useful for resource pools, bounded collections, and rate limiters.

Surprise: permits aren't tied to threads. Thread A can acquire a permit and thread B can release it. The semaphore isn't even limited to its initial count — you can release more than were initially issued. Think of acquire as "consume" and release as "create."
How does BoundedHashSet use a semaphore to bound a collection?
Construct a Semaphore(bound) alongside a plain (thread-safe) Set. On add: acquire a permit first, then add — if the element was a duplicate (add returned false), immediately release the permit. On remove: if removal succeeded, release a permit.

The underlying set knows nothing about the bound — the semaphore handles capacity entirely.
How does CyclicBarrier work, and what is the optional "barrier action"?
CyclicBarrier(n): threads call await(), blocking until all n parties arrive. Then all are released, the barrier resets, and the cycle can repeat. If await times out or is interrupted, the barrier breaks and all pending awaits throw BrokenBarrierException.

The barrier action is an optional Runnable passed to the constructor, executed on one of the subtask threads right before parties are released. Perfect for "commit the results of this step" between iterations.
Why does CellularAutomata use Ncpu worker threads and a CyclicBarrier?
For CPU-bound problems with no I/O and no shared data, Ncpu (or Ncpu + 1) threads yield optimal throughput — more threads just compete for CPU and degrade performance.

The barrier enforces the step boundaries: every worker must finish step k for its sub-board before anyone starts step k+1. The barrier action commits the new values between steps, so all threads observe a consistent board.
Memoizer1 (HashMap + synchronized compute): what's wrong with it?
It's correct, but it serializes every call to compute. A queue of threads asking for different values all wait behind one slow computation.

In the worst case, caching makes things slower than no cache — a scalability bottleneck, not a scalability win.
Memoizer2 (ConcurrentHashMap, cache values): what's the window of vulnerability?
Two threads asking for the same key at the same time both see no entry, both start computing, and both do the expensive work.

For memoization this is merely inefficient. For a cache meant to guarantee once-and-only-once initialization (object caches), it's a safety bug — you get two distinct instances.
Why does Memoizer3 cache Future<V> instead of V, and what tiny window of vulnerability remains?
Caching the in-progress Future lets late arrivers wait on the existing computation via Future.get() instead of starting another. Massive reduction in duplicate work.

Remaining defect: the get then put on the cache is still a non-atomic check-then-act. Two threads can both see null, both create a FutureTask, and both put — so both end up running the same computation.
How does the final Memoizer close the check-then-act window, and what is cache pollution?
Use ConcurrentMap.putIfAbsent — an atomic insertion. Exactly one FutureTask wins the race; losers get the winner's Future and call get() on it.

Cache pollution: if a cached Future completes exceptionally or is cancelled, every future lookup inherits that failure. Memoizer wraps the logic in while (true) and on CancellationException removes the poisoned Future from the cache and retries. You might similarly evict on RuntimeException if retrying could succeed.
