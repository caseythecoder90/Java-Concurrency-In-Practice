Besides atomicity, what second major problem does synchronized solve?
Memory visibility. When thread A releases a lock and thread B later acquires the same lock, everything A did before the release is guaranteed visible to B after the acquire.

Without synchronization, there is no guarantee a reader will ever see another thread's writes — on any timely basis, or at all.
What are the three possible outcomes of running NoVisibility (Listing 3.1)?
The main thread writes number = 42; ready = true;; the reader spins on while (!ready) and prints number.

1. Prints 42 (the happy path).
2. Prints 0 — the write to ready becomes visible before the write to number due to reordering.
3. Loops forever — the write to ready never becomes visible to the reader.
What is stale data, and why is it especially dangerous?
A stale value is an old value still visible to a reader after another thread has updated the variable.

It's dangerous because staleness is not all-or-nothing — a thread can see a fresh value for one field and a stale value for another field written before it, leaving objects in observably inconsistent states.
What's special about non-volatile long and double variables across threads?
The JVM is allowed to split a 64-bit read or write into two 32-bit operations. A reader can see a "torn" value — the high 32 bits from one write and the low 32 bits from another, a value no thread ever actually wrote.

Shared mutable long / double must be declared volatile or guarded by a lock.
What visibility guarantee does volatile give that regular variables don't?
A read of a volatile always returns the most recent write by any thread — the value is never cached in a register or per-CPU cache where it's hidden from other processors.

Writing a volatile also acts like exiting a synchronized block: everything visible to the writer before the volatile write becomes visible to a thread that reads the volatile. Reading one is like entering a synchronized block.
What can volatile not do that synchronized or atomic classes can?
Atomicity of compound actions. count++ on a volatile is still three operations (read, add, write); interleaved threads can lose updates.

"Locking can guarantee both visibility and atomicity; volatile variables can only guarantee visibility."
Use AtomicLong or a lock for read-modify-write.
When is it appropriate to use volatile?
When all three hold:

1. Writes don't depend on the current value (or only one thread ever writes).
2. The variable doesn't participate in invariants with other state variables.
3. Locking isn't required for any other reason.

Classic use: a completion/shutdown/status flag like volatile boolean asleep.
What does it mean for an object to "escape"?
It's been made reachable outside its intended scope — stored in a public static field, returned from a non-private method, passed to an alien method, exposed via an inner class, etc.

Once escaped, you must assume other code may use or misuse it concurrently. This is why encapsulation matters for concurrency.
What is an "alien method" and why does it matter for publication?
An alien method is any method whose behavior isn't fully specified by your class — methods on other classes, or overridable (neither private nor final) methods in your own class.

Passing an object to an alien method is publication, because you can't know whether it will retain or leak the reference to another thread.
Why is ThisEscape (Listing 3.7) broken, and how does SafeListener fix it?
Registering an anonymous inner-class listener from the constructor leaks this — the inner class holds an implicit outer reference, so other threads can reach the ThisEscape object before construction finishes and observe a partially constructed instance.

SafeListener uses a private constructor + public static factory method that only registers the listener after the object is fully constructed.
What is thread confinement, and why does it make non-thread-safe objects safe?
Thread confinement means an object is only accessed by a single thread. With no sharing, there are no race conditions or visibility issues — even a non-thread-safe object is safe to use this way.

Real-world examples: Swing components confined to the event dispatch thread; pooled JDBC Connections confined to a thread for the duration of a request.
What's the difference between ad-hoc, stack, and ThreadLocal confinement?
Ad-hoc: confinement enforced purely by convention/documentation — fragile; nothing in the language helps.

Stack: object only reachable via local variables; the stack itself enforces it. Primitives are always stack-confined (you can't take a reference to one).

ThreadLocal: ThreadLocal<T> gives each thread its own copy via get/set; values are stored on the Thread object itself.
Give a realistic use case for ThreadLocal.
Associating a per-thread database connection so code doesn't have to pass Connection through every method call (Listing 3.10), or storing transaction context in a J2EE container.

Each thread's first get calls initialValue; subsequent calls return that thread's stored value.
What's the downside of ThreadLocal?
It behaves like a global variable or hidden method parameter, creating invisible coupling between classes. Code using a ThreadLocal is implicitly tied to whatever framework populates it, and the coupling isn't visible in method signatures.

Use with care — prefer passing values explicitly when you can.
What are the three requirements for an object to be immutable?
1. Its state cannot be modified after construction.
2. All its fields are final.
3. It is properly constructed — the this reference does not escape during construction.

Note: final fields alone aren't sufficient, because a final field can reference a mutable object.
Can an immutable object use mutable objects internally?
Yes. ThreeStooges (Listing 3.11) uses a mutable HashSet internally.

As long as the mutable state is (a) established entirely in the constructor, (b) never exposed, and (c) never modified afterward, the containing object is immutable.
How do OneValueCache and VolatileCachedFactorizer (Listings 3.12-3.13) solve the two-field atomicity problem without locking?
OneValueCache is an immutable holder containing both the number and its factors — any thread that sees the reference sees a consistent snapshot.

VolatileCachedFactorizer keeps a single volatile OneValueCache cache. Updating is a one-shot reassignment of the volatile field, which becomes instantly visible to all threads. No lock needed.
Why is this unsafe publication? public Holder holder; holder = new Holder(42);
Without synchronization, other threads may:

(a) see a stale value for holder (null or earlier), or
(b) see the reference but see Holder's fields in a partially constructed state — e.g. the default 0 instead of the constructor-set value.

Holder.assertSanity's if (n != n) check really can fire under improper publication.
What are the four safe publication idioms for a properly constructed object?
1. Initialize it from a static initializer.
2. Store the reference in a volatile field or AtomicReference.
3. Store the reference in a final field of a properly constructed object.
4. Store the reference in a field properly guarded by a lock.
Do thread-safe collections provide safe publication?
Yes. Placing an object into Hashtable, ConcurrentMap, synchronizedMap, Vector, CopyOnWriteArrayList/Set, synchronizedList/Set, BlockingQueue, or ConcurrentLinkedQueue safely publishes it to any thread that retrieves it later.

No extra synchronization is needed at the handoff — the collection's internal synchronization covers you.
What is an effectively immutable object?
An object that isn't technically immutable (it could be mutated), but by convention is never modified after publication.

Once safely published, an effectively immutable object can be shared across threads without further synchronization. Classic example: Date values placed in a synchronizedMap that no one mutates afterward.
What special guarantee does the JMM give to truly immutable objects?
Initialization safety — they can be safely accessed by any thread even if synchronization was not used to publish them.

This only applies when all immutability requirements are met (unmodifiable state, all fields final, proper construction). A non-immutable object published unsafely is not covered.
What are the four sharing policies the book recommends documenting for any shared object?

Every shared reference should come with a known policy — document it.
How do publication requirements differ for immutable, effectively immutable, and mutable objects?
Immutable: can be published through any mechanism, even a data race.

Effectively immutable: must be safely published; then no further synchronization is needed.

Mutable: must be safely published and be either thread-safe or guarded by a lock on every access.
