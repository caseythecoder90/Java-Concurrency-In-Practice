Chapter 7: Cancellation and Shutdown

1. Why is Java's cancellation cooperative rather than preemptive, and what mechanism does it use?
Java provides no safe way to forcibly stop a thread — Thread.stop and Thread.suspend were deprecated for exactly this reason. Forcible termination can leave shared data structures with broken invariants (half-updated maps, half-written log lines, half-committed transactions).

Instead Java offers interruption: one thread can ask another to stop, and the recipient chooses when and how to honor the request. The task knows best what cleanup it needs — the canceller doesn't.

Concretely, a thread has an interrupted status flag; Thread.interrupt() sets it; Thread.isInterrupted() reads it; many blocking library methods throw InterruptedException in response. None of this stops the thread by itself — it just delivers a message.
2. What is a "cancellation policy" and why does the chapter argue every cancellable task needs one?
A written description of how, when, and what:

• How external code requests cancellation (a method? Interruption? A flag?)
• When the task checks for cancellation (every iteration? Every blocking call?)
• What actions the task takes when it detects cancellation (rollback? Save state? Just exit?)

Without a written policy, callers and tasks make incompatible assumptions and you get races, leaks, and stuck threads. The book uses the bank stop-payment analogy: real cancellation has rules, responsiveness guarantees, and follow-up actions.
3. When does the simple volatile boolean cancelled flag pattern work, and when does it break?
Works: tight CPU-bound loops where the task hits the flag check often. 
while (!cancelled) {
    p = p.nextProbablePrime();
    primes.add(p);
}
Breaks: the moment the loop calls a blocking method that doesn't check the flag. BrokenPrimeProducer calls queue.put(...) — if the queue is full and the consumer stops, the producer wedges in put and never re-checks cancelled. The flag is set forever and ignored forever.

Fix: use interruption, which most blocking methods do respect.
4. What is the interruption API on Thread, and what's the trap with the static interrupted()? 
public class Thread {
    public void interrupt();              // set the flag
    public boolean isInterrupted();       // read the flag (non-destructive)
    public static boolean interrupted();  // read AND CLEAR the flag
}
Thread.interrupted() is the trap: it both reads and clears. If you call it and don't propagate or act on the value, you've silently swallowed the interrupt — everything upstream loses the signal.

Default to Thread.currentThread().isInterrupted(); only use the static version when you specifically intend to consume the signal.
5. What does Thread.interrupt() actually do?
It sets the interrupted flag. That's it. The thread doesn't stop running.

What happens next depends on what the thread is doing:

• Blocked in an interruptible method (sleep, wait, take, put, get, NIO ops): the method clears the flag and throws InterruptedException.
• Running normal code: nothing immediate. Flag stays set. The thread sees it next time it polls.
• Blocked on socket I/O or a synchronized lock: the flag gets set but the block continues. Interruption has no effect on these.

Reading "interrupt" as "forcibly stop" is a common misconception — it's just a message.
6. What's the difference between a task's cancellation policy and a thread's interruption policy?
Cancellation policy — what the task does when asked to stop. Owned by the task author.

Interruption policy — what the thread does when its interrupt flag is set. Owned by the thread's owner (usually a thread pool).

One interrupt may have two recipients. Interrupting a pool worker can mean "cancel the current task" AND "shut down the worker thread." A task on a borrowed thread should preserve the flag (e.g. by re-throwing or restoring it) so the thread's owner can act on it too.
"You should not interrupt a thread unless you know what interruption means to that thread." (p. 89)
7. What are your two options when you catch InterruptedException, and what's the third option that's almost always wrong?
1. Propagate. Add throws InterruptedException to your signature and let the exception bubble up. 
public Task getNextTask() throws InterruptedException {
    return queue.take();
}
2. Restore and continue. When you can't propagate (you're inside Runnable.run), preserve the signal: 
catch (InterruptedException e) {
    Thread.currentThread().interrupt();
}
3. (Usually wrong) Swallow. Catch and do nothing. Only acceptable if you are the thread's owner and about to exit. General-purpose task and library code should never swallow.
8. If a method is not cancellable but calls interruptible blocking methods, how should it handle InterruptedException?
Save the interruption locally, retry the blocking call, and restore the flag at the very end in finally — not immediately on catch. 
boolean interrupted = false;
try {
    while (true) {
        try { return queue.take(); }
        catch (InterruptedException e) {
            interrupted = true;   // remember, retry
        }
    }
} finally {
    if (interrupted) Thread.currentThread().interrupt();
}
Restoring the flag too early would cause take() to immediately re-throw — an infinite loop, since most interruptible methods check the flag on entry.
9. Why is scheduling an interrupt against the calling thread (Listing 7.8 timedRun) considered "Don't Do This"?
It violates the rule "don't interrupt a thread you don't own." timedRun can be called from any thread — you don't know that thread's interruption policy.

Worse, if the task finishes before the timeout, the scheduled interrupt fires after timedRun has already returned. The interrupt now hits whatever code is running next on the caller's thread — arbitrary code, in an arbitrary state. Result: bizarre, hard-to-reproduce bugs.

The right answer is to run the task on a thread you own (Listing 7.9) or, better, use a Future (Listing 7.10).
10. What does Future.cancel(boolean mayInterruptIfRunning) do, and when is each value appropriate?
Returns whether the cancellation was delivered, not whether the task honored it.

cancel(true) — if the task is running, interrupt the worker thread. Use this when the task knows how to respond to interruption.

cancel(false) — "don't run this if it hasn't started yet." Use for tasks that are not designed to handle interruption mid-flight.

Calling cancel on a completed task is harmless — that's why the canonical pattern places task.cancel(true) unconditionally in a finally block.
11. Why is it safe to call Future.cancel(true) on tasks running in a standard Executor?
Because the standard Executor implementations install an interruption policy on their worker threads that explicitly supports task cancellation. Interrupting a pool worker means "cancel the current task," and the pool replaces the worker if needed.

It is not safe to interrupt a pool thread directly — you don't know what task is running, and you might cancel an unrelated piece of work. Always cancel through the Future.

Implication: code your tasks to treat interruption as cancellation. Then you can cancel them safely through their Futures, regardless of which Executor runs them.
12. What's the canonical timed-run pattern using Future (Listing 7.10)? 
public static void timedRun(Runnable r, long timeout, TimeUnit unit)
        throws InterruptedException {
    Future<?> task = taskExec.submit(r);
    try {
        task.get(timeout, unit);
    } catch (TimeoutException e) {
        // task will be cancelled below
    } catch (ExecutionException e) {
        throw launderThrowable(e.getCause());
    } finally {
        task.cancel(true);   // harmless if completed
    }
}
Submit, wait with deadline, unwrap on error, always cancel. Cancelling a completed task is a no-op, so the finally block can run unconditionally.

14. What's the ReaderThread pattern (Listing 7.11) and why is it useful?
Encapsulate nonstandard cancellation by overriding interrupt: 
public class ReaderThread extends Thread {
    private final Socket socket;
    public void interrupt() {
        try { socket.close(); }
        catch (IOException ignored) { }
        finally { super.interrupt(); }
    }
    public void run() {
        try {
            while (true) {
                int n = in.read(buf);  // not interruptible
                if (n < 0) break;
                processBuffer(buf, n);
            }
        } catch (IOException e) { /* exit */ }
    }
}
Calling interrupt() now does both things: closes the socket (unblocks the read) and sets the standard flag. The thread becomes responsive to interruption even though the underlying I/O isn't.
15. What is newTaskFor for, and how does CancellingExecutor use it?
ThreadPoolExecutor.newTaskFor(Callable<T>) is a hook that returns the RunnableFuture the executor uses for a submitted task. By overriding it you can give a task its own custom Future — whose cancel() can do extra work like closing a socket or logging metrics. 
protected <T> RunnableFuture<T> newTaskFor(Callable<T> callable) {
    if (callable instanceof CancellableTask)
        return ((CancellableTask<T>) callable).newTask();
    return super.newTaskFor(callable);
}
It's the task-side equivalent of ReaderThread: encapsulate nonstandard cancellation, but on the Future instead of the Thread.
16. Why is the naïve shutdownRequested-flag in LogWriter (Listing 7.14) unreliable?
It's a check-then-act race: 
public void log(String msg) throws InterruptedException {
    if (!shutdownRequested) queue.put(msg);   // race
    else throw new IllegalStateException();
}
A producer can read shutdownRequested == false, get pre-empted, and the consumer can shut down before the producer reaches put. The producer then wedges on put with no one ever draining the queue.

Fix (Listing 7.15): hold a lock for the check + reservation counter, then call put outside the lock. The consumer waits until isShutdown AND reservations == 0 before exiting.
17. When and how do poison pills work for shutting down a producer-consumer service?
Place a recognizable sentinel on the queue meaning "stop after this": 
private static final File POISON = new File("");
// consumer
File f = queue.take();
if (f == POISON) break;
Requirements:
• Unbounded queue — otherwise the producer can block on put before enqueueing the pill.
• Known producer/consumer counts — for N producers, the consumer waits until it sees N pills.

Generalises to multiple consumers (each producer enqueues Nconsumers pills), but it gets unwieldy. For more flexible shutdown, ExecutorService + shutdown/shutdownNow is usually simpler.
18. What's the "one-shot execution service" pattern (Listing 7.20) and why is it useful?
Create a private ExecutorService whose lifetime is bounded by a single method call: 
boolean checkMail(Set<String> hosts, long timeout, TimeUnit unit)
        throws InterruptedException {
    ExecutorService exec = Executors.newCachedThreadPool();
    AtomicBoolean hasNewMail = new AtomicBoolean();
    try {
        for (String host : hosts)
            exec.execute(() -> { if (checkMail(host)) hasNewMail.set(true); });
    } finally {
        exec.shutdown();
        exec.awaitTermination(timeout, unit);
    }
    return hasNewMail.get();
}
The executor is born and dies inside the method. The caller sees a method returning a value — no service lifecycle to manage. Great for fan-out / fan-in batches.
19. What does shutdownNow tell you, what does it not tell you, and how does TrackingExecutor fix the gap?
Tells you: the list of tasks that were submitted but never started.

Doesn't tell you: which tasks started but didn't finish — you just know the workers were interrupted.

TrackingExecutor trick: wrap each submission to record any task that returns from run() while (a) the executor is shutting down and (b) the worker's interrupt flag is set: 
finally {
    if (isShutdown() && Thread.currentThread().isInterrupted())
        cancelledTasks.add(runnable);
}
Has a small race — a task that finishes its last instruction just before the interrupt is recorded as cancelled. Tolerable when tasks are idempotent (re-running them is safe), as in a web crawler.
20. Why is a thread that dies from an uncaught RuntimeException particularly dangerous in a server, and what's the canonical structural fix?
It's silent. A pool quietly shrinks by one worker. The application keeps "working" with reduced capacity. Eventually capacity hits zero or behavior degrades in confusing ways — far away from the original throw.

Structural fix (Listing 7.23): wrap untrusted task code in try/catch (Throwable)/finally inside the worker loop, and notify the framework when a thread exits abnormally: 
public void run() {
    Throwable thrown = null;
    try {
        while (!isInterrupted())
            runTask(getTaskFromWorkQueue());
    } catch (Throwable e) {
        thrown = e;
    } finally {
        threadExited(this, thrown);
    }
}
ThreadPoolExecutor uses exactly this pattern, which is why a throwing task doesn't kill its pool.
21. What is UncaughtExceptionHandler, and what's the gotcha about execute vs submit? 
public interface Thread.UncaughtExceptionHandler {
    void uncaughtException(Thread t, Throwable e);
}
Set per-thread (setUncaughtExceptionHandler) or globally (setDefaultUncaughtExceptionHandler). For pools, supply a ThreadFactory that installs the handler on each worker.

Gotcha: the handler fires only for tasks submitted via execute. Tasks submitted via submit capture the throwable inside their Future — Future.get() throws ExecutionException wrapping the cause. Same exception, two completely different paths.

Always at least log with a UEH; don't rely on stack traces being seen.
22. What are shutdown hooks, and what are the three rules they need to follow?
Threads registered with Runtime.addShutdownHook(thread). Run during orderly JVM shutdown (last non-daemon exits, System.exit, SIGINT). They do not run on abrupt shutdown (Runtime.halt, SIGKILL).

Rules:
• Thread-safe — they run alongside any application threads still alive.
• Defensive — make no assumptions about other hooks or the application state.
• Fast — they delay JVM exit while the user is waiting.
• Independent — hooks run concurrently in arbitrary order. A hook that closes the log file may break a hook that tries to log.

Best practice when you have several things to clean up: register one ordering hook that calls them sequentially.
23. What are daemon threads and when should you use them?
Threads that don't keep the JVM alive. Marked with thread.setDaemon(true) before start(). Daemon-ness is inherited from the creating thread. The JVM exits when no normal threads are left, abandoning daemons mid-execution — finally blocks don't run, stacks don't unwind.

Use sparingly — only for stateless housekeeping: cache eviction, idle pings, GC threads. Never for I/O — your write may be cut off mid-stream.
"Daemon threads are not a good substitute for properly managing the lifecycle of services within an application." (p. 103)
24. Why does the book say "Avoid finalizers"?
Object.finalize sounds useful (releasing native handles when the GC reclaims) but is broken in practice:

• No timing guarantees — may run any time, may never run at all if the JVM exits.
• Run on a JVM-managed thread — any state they touch needs synchronization.
• Significant GC overhead — objects with finalizers cost more to collect.
• Hard to write correctly — resurrection, exception swallowing, ordering all become traps.

Use try-with-resources / explicit close instead. For native handles, Cleaner (Java 9+) is far safer.
