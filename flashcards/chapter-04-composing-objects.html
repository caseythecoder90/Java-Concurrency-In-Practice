What are the three steps in the book's design process for a thread-safe class?
1. Identify the state variables that form the object's state.
2. Identify the invariants that constrain those variables.
3. Establish a synchronization policy — which locks guard which variables, what combination of immutability/confinement/locking maintains thread safety.

And document the policy.
What is a "synchronization policy"?
The design rules for how an object coordinates access to its state:

• Which fields are guarded by which locks
• Which fields are immutable or thread-confined
• Which operations must be atomic

It belongs in comments and @GuardedBy annotations — not just in your head.
How do invariants and post-conditions create synchronization requirements?
Invariants (valid states) force you to encapsulate the state so clients can't put it in an invalid state.

Post-conditions (valid state transitions) force compound actions to be atomic — you can't release the lock between steps, or another thread may observe an illegal intermediate state.
What is the Java monitor pattern?
An object encapsulates all its mutable state and guards it with the object's own intrinsic lock. All access goes through synchronized methods.

Used by Vector, Hashtable, Counter. The primary advantage is simplicity — synchronization logic lives in one place.
Why is using a private Object lock often better than using this as the intrinsic lock?
A private lock is encapsulated: client code cannot acquire it, so clients can't accidentally participate in (or break) your synchronization policy or cause liveness problems.

Verifying the locking policy requires examining only one class instead of the whole program.
What is instance confinement?
Encapsulating a non-thread-safe object entirely within a thread-safe class, where the inner object never escapes and all access goes through the wrapper's synchronized methods.

PersonSet's private HashSet is instance-confined. This is also how Collections.synchronizedList works — the Decorator pattern.
What is "state ownership" and why does it matter for thread safety?
Ownership isn't a language construct — it's a design concept. The owner of a state variable decides its locking protocol.

Encapsulation typically implies ownership. Split ownership (e.g. ServletContext attributes) is common: the container owns the map, the application owns the values, and each side is responsible for its share of the safety work.
What is delegation of thread safety?
The composite class relies on thread-safe underlying components to provide its thread safety, and adds no synchronization of its own.

Examples: CountingFactorizer delegates to AtomicLong; DelegatingVehicleTracker delegates to ConcurrentHashMap + immutable Point.
What's the rule for when a composite can safely delegate to multiple thread-safe components?
The underlying components must be independent:

• The composite imposes no invariants involving multiple state variables, and
• No operation has invalid state transitions.

VisualComponent (two unrelated listener lists) qualifies. NumberRange (lower ≤ upper) does not.
Why isn't NumberRange (Listing 4.10) thread-safe even though it uses AtomicIntegers?
setLower and setUpper are check-then-act sequences without sufficient locking.

Given range (0, 10), concurrent setLower(5) and setUpper(4) can both pass their checks and both write, producing (5, 4) — an invalid state.

When variables participate in an invariant, per-variable atomics aren't enough — the compound action needs a single lock.
When is it safe to publish an underlying state variable directly to clients?
When all three hold:

1. The variable is thread-safe.
2. It doesn't participate in invariants that constrain its value.
3. No operation on it has prohibited state transitions.

Counter.value fails (2). VisualComponent.mouseListeners passes all three.
What problem does SafePoint.get() returning int[]{x, y} solve?
It provides an atomic combined read of both coordinates.

Separate getX()/getY() would let another thread mutate the point between the two calls, so a caller could observe an (x, y) pair the vehicle was never at. Related values must be fetched atomically.
What is the private constructor capture idiom, and why does SafePoint use it?
A naive copy constructor this(p.x, p.y) would race between the two reads.

SafePoint instead has a private constructor taking int[]; the public copy constructor calls this(p.get()), doing a single atomic snapshot of both coordinates before passing them to the private one.
What's the difference between returning an unmodifiable live view vs. a snapshot of a map?
Live view — Collections.unmodifiableMap(backingMap): read-only but reflects ongoing updates. Fresh data, but can look inconsistent if callers iterate while it mutates.

Snapshot — Collections.unmodifiableMap(new HashMap<>(backingMap)): a point-in-time copy. Consistent but goes stale.

Pick deliberately.
What are the four strategies for adding a new atomic operation to an existing thread-safe class, safest to most fragile?
1. Modify the original class — safest; policy stays in one place.
2. Extend the class (BetterVector) — fragile; locking code now spans files.
3. Client-side locking — even more fragile; locking code in an unrelated class.
4. Composition (ImprovedList) — wrap with your own lock; independent of the underlying strategy.

Composition is the preferred approach.
Why does the naive ListHelper in Listing 4.14 fail to make putIfAbsent atomic?
It synchronizes on the wrong lock — the helper itself, not the underlying list.

The list's own synchronized methods use the list's intrinsic lock; the helper's synchronized blocks don't interact with it, so other threads can modify the list between the contains check and the add.
How does client-side locking work, and what's its main caveat?
Acquire the same lock the target object uses (e.g. synchronized (list) for a Collections.synchronizedList) and perform your compound action inside.

Caveat: it only works if the target class documents that it supports client-side locking on its intrinsic lock. Without that commitment, you're guessing at its internal policy and the next release could silently break you.
Why is composition the preferred way to add functionality to an existing thread-safe class?
The composition wrapper uses its own lock to synchronize every delegated method, making it independent of the underlying class's locking strategy.

It's effectively the Java monitor pattern over an existing object — works whether the underlying object is thread-safe or not, and doesn't break if the underlying class changes its policy.
What should you document about a thread-safe class, and for whom?
"Document a class's thread safety guarantees for its clients; document its synchronization policy for its maintainers."
Clients need: is it thread-safe? Does it make callbacks with a lock held? Which lock (if any) supports client-side locking?

Maintainers need: the synchronization policy — which fields are @GuardedBy which lock, which invariants hold, which operations are atomic.
Which of ServletContext, HttpSession, DataSource, SimpleDateFormat, and JDBC Connection are thread-safe?
Thread-safe (by reasonable inference — containers access them concurrently, no client-side locking in spec examples): ServletContext, HttpSession, DataSource.

Your responsibility: attributes stored in ServletContext/HttpSession — make them thread-safe or effectively immutable.

NOT thread-safe: SimpleDateFormat (classic gotcha).

Not intended to be shared: JDBC Connection — typically thread-confined per request.
