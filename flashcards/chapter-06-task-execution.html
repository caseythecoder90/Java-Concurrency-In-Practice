Chapter 6: Task Execution

1. What makes a good "task" in the context of an executor framework?
A self-contained unit of work that is independent of other tasks — no shared mutable state with siblings, doesn't depend on their results, and can be scheduled on any worker thread.

Granularity matters: too small and dispatch overhead dominates; too large and you lose parallelism. Small to medium-sized is the sweet spot.

Independence is what lets the framework choose the execution policy — pool size, ordering, queueing — without breaking correctness.
2. Why is a sequential server ("accept then handleRequest in the same thread") almost never acceptable in production?
While handleRequest runs, accept() is not called — new connections sit in the OS backlog or are refused. Throughput is roughly 1 / mean(handleRequest) requests per second. The CPU is idle most of the time waiting on I/O, and clients see arbitrary queue delays.

Fine for demos and single-user tools; unacceptable for any server with concurrent clients.
3. What three problems does the naïve thread-per-task approach run into?
1. Thread lifecycle overhead. Creating and destroying threads has measurable startup latency and per-thread memory cost (~512KB-1MB stack on 64-bit).

2. Resource consumption. Active threads consume kernel scheduling slots and stacks. More threads than cores adds context-switching overhead and GC pressure.

3. Stability. JVM/OS thread limits cause OutOfMemoryError ("unable to create native thread") when exceeded. The whole JVM — not just the request handler — goes unstable.

Throughput plateaus then degrades as thread count grows past available cores.
4. What does the Executor interface look like, and why is such a tiny interface so important? 
public interface Executor {
    void execute(Runnable command);
}
One method, one parameter. Its power comes from decoupling task submission from task execution: callers say "run this," and a single configuration point decides where, when, in what thread, and with what queueing/rejection rules.

It also formalises the producer-consumer shape: submitters produce tasks, worker threads consume them, the executor mediates with an internal queue.
5. What is an "execution policy" and why does the chapter argue you should make it explicit?
The bundle of decisions a thread pool/executor makes:

• In what thread will tasks run?
• In what order? (FIFO, LIFO, priority)
• How many tasks may run concurrently?
• How many may queue?
• What happens when overloaded? (reject, drop oldest, run-in-caller, ...)
• Pre/post hooks per task?

Putting all of these in one place — an Executor — means you can tune them without rewriting calling code.

7. Why is newCachedThreadPool dangerous as a default for a server?
It has no upper bound on threads. Under sustained burst load it keeps creating workers — you can hit OS thread limits and OOM, the same failure mode as raw thread-per-task.

It uses a SynchronousQueue internally, so a submitted task either grabs an idle thread immediately or causes a new thread to be created. There is no queue to absorb bursts.

Use it for short-lived prototypes; for production servers, pick a fixed pool or a hand-tuned ThreadPoolExecutor with a bounded queue.
8. Why does the Executor interface need ExecutorService on top of it?
Executor says nothing about lifecycle — but the JVM won't exit while non-daemon threads are alive, so a pool that doesn't shut down hangs your process.

ExecutorService adds: 
void shutdown();              // graceful
List<Runnable> shutdownNow(); // forceful
boolean isShutdown();
boolean isTerminated();
boolean awaitTermination(long, TimeUnit);
<T> Future<T> submit(Callable<T>);
Three states: running → shutting down → terminated.
9. What is the difference between shutdown() and shutdownNow()?
shutdown() — "stop accepting work, but finish what's been submitted." New submissions throw RejectedExecutionException; queued and running tasks complete normally.

shutdownNow() — "I need to stop fast." Drains the queue (returning unstarted tasks) and interrupts running threads. Whether the running tasks actually stop is up to them — interruption is cooperative (they must respond to it).

Common pattern: shutdown(), wait via awaitTermination, fall back to shutdownNow() if the wait expires.
10. What's the canonical "graceful then forceful" shutdown idiom? 
exec.shutdown();
try {
    if (!exec.awaitTermination(30, TimeUnit.SECONDS)) {
        exec.shutdownNow();
        if (!exec.awaitTermination(30, TimeUnit.SECONDS))
            log.warn("executor did not terminate");
    }
} catch (InterruptedException e) {
    exec.shutdownNow();
    Thread.currentThread().interrupt();
}
Give running tasks a fair chance to complete; force after the deadline; if you yourself are interrupted while waiting, escalate to shutdownNow and restore the flag.
11. Why does the book argue you should never use java.util.Timer in new code?
Two failure modes:

1. Single thread. A long task delays every other timer task, and periodic tasks may overlap their period.

2. Unchecked exceptions kill the timer. If a task throws, the entire Timer thread dies. Pending tasks never run, and there's no notification. The timer looks alive but isn't — a silent failure.

Use ScheduledThreadPoolExecutor instead: multiple threads (one bad task doesn't starve others) and exceptions surface via Future.
12. How does OutOfTime (Listing 6.9) demonstrate the Timer problem?
It schedules two tasks — one at +1s that throws RuntimeException, one at +1s that just prints. Run it and you see:

• The throwing task fires.
• The thread dies.
• The second task never runs.
• cancel() on the timer throws IllegalStateException ("timer already cancelled") — and the JVM's main thread reports the exception only on shutdown.

A real outage scenario for anyone running scheduled jobs on a Timer.
13. What is Callable and why is it more useful than Runnable for real work? 
public interface Callable<V> {
    V call() throws Exception;
}
Two things Runnable can't do:

1. Returns a value. Real work usually has a result; Runnable forces you to plumb it through shared state.

2. Throws checked exceptions. Runnable.run can only let unchecked exceptions escape, which often forces awkward wrapping.

Submit a Callable to an ExecutorService and you get a Future<V> back — result and error in one object.
14. What does Future<V>.get() do and what exceptions can it throw?
Blocks until the task completes, then returns its result. The exceptions surface what happened to the task:

InterruptedException — the waiter was interrupted.
ExecutionException — the task threw; e.getCause() is the original throwable.
CancellationException — the task was cancelled before completing.
TimeoutException — only on the timed overload, when the wait expires.

Always inspect e.getCause() to recover the real error — ExecutionException itself is the wrapper, not the bug.
15. What is launderThrowable and why is it useful?
A helper that unwraps the Throwable inside an ExecutionException and rethrows it cleanly:

• If it's a RuntimeException, throw it as-is.
• If it's an Error, throw it as-is.
• Otherwise it's a checked exception that shouldn't be there — throw IllegalStateException wrapping it ("not a checked exception").

Always handle known checked exceptions before calling launderThrowable; this method is the catch-all for unexpected ones. Same helper appears in chapter 5's Memoizer.
16. Why does the sequential page renderer have terrible UX?
Page-load latency = sum of all image download times, plus text rendering.

The CPU sits idle most of the time waiting on network. Nothing renders until everything is downloaded. The user stares at a blank page until the slowest image arrives.

It's a textbook case of I/O-bound work waiting in series when it could be in parallel — trivially exploitable parallelism left on the floor.
17. What does the Future-based renderer (Listing 6.12) buy you, and what's still wrong with it?
Buys: the main thread renders text immediately while a background task downloads images. Text appears fast; user sees something.

Still wrong: all image downloads are inside one submitted task and run serially. After the text appears, the user waits for the slowest image before any image renders. They arrive as a bundle, not as they finish.

The next refactor is per-image submission via CompletionService.
18. What is CompletionService and what problem does it solve?
It wraps an Executor and adds an internal BlockingQueue of completed Futures. 

cs.submit(task);              // submit normally
Future<V> f = cs.take();      // block for next FINISHED
Tasks come back in order of completion, not submission. The natural pattern for "do N independent things and process each result as soon as it's ready" — like the page renderer that paints each image the moment its download finishes.

Without it, you'd be polling future.isDone() in a loop — busy-wait or arbitrary-delay city.
19. What's the basic CompletionService usage pattern? 
CompletionService<V> cs = new ExecutorCompletionService<>(exec);
for (Task t : tasks) cs.submit(t);
for (int i = 0; i < tasks.size(); i++) {
    try {
        V result = cs.take().get();
        process(result);
    } catch (ExecutionException e) {
        throw launderThrowable(e.getCause());
    }
}
Submit all, take exactly N times. Don't track Futures yourself unless you need to cancel a specific one.
20. Why is calling future.cancel(true) after a TimeoutException not optional?
A TimeoutException from Future.get(timeout, ...) means "the waiter gave up," not "the task stopped." The task keeps running, holds whatever resources it had, and produces a result no one will read.

For long-running or expensive tasks that's a leak. The idiom: 
try {
    return f.get(timeout, MILLISECONDS);
} catch (TimeoutException e) {
    return defaultValue;
} finally {
    f.cancel(true);
}
Cancellation is still cooperative — the task must respond to interruption — but at least you've requested it.
21. What does invokeAll(tasks, timeout, unit) guarantee, and how do you read its results?
Submits all tasks, blocks up to the timeout, then returns a list of Futures in submission order (not completion order). Any task that didn't finish in time is cancelled; its Future has isCancelled() == true.

Read each result like: 
for (Future<V> f : futures) {
    if (f.isCancelled()) handleTimeout();
    else try { handleResult(f.get()); }
         catch (ExecutionException e) { handleError(e.getCause()); }
}
Canonical use: travel-portal — ask N airlines, give them all 1 second, return what came back.
22. When should you choose CompletionService versus invokeAll with a timeout?
CompletionService — you want to process each result as it arrives, possibly do partial work before all are done, and you don't necessarily care about a hard deadline. Order is completion order.

invokeAll(timeout) — you want a single batch with a hard deadline. After the deadline you stop waiting and treat unfinished tasks as failures. Order is submission order, with cancelled futures interspersed.

The page renderer fits the first; the travel portal fits the second.
23. What's the practical difference between execute(Runnable) and submit(Callable) for error handling?
execute(Runnable) — uncaught exceptions go to the thread's UncaughtExceptionHandler (and to ThreadPoolExecutor.afterExecute). Easy to lose track of in a server log.

submit(...) — returns a Future. Any thrown exception is captured and re-thrown wrapped in ExecutionException when you call get(). Errors are visible by default to whoever holds the future.

Prefer submit when you have a caller that wants the result anyway; otherwise wire up afterExecute or a UncaughtExceptionHandler.
24. Why is newFixedThreadPool's default queue still a footgun?
Internally it uses an unbounded LinkedBlockingQueue. The thread count is bounded, but the work backlog isn't — if producers outpace workers, the queue grows until you OOM.

The "cap on threads" is the easy part; bounding the queue is the part that gives you actual backpressure. For real overload protection, build your own ThreadPoolExecutor with a bounded queue and a sensible RejectedExecutionHandler (covered in chapter 8).
